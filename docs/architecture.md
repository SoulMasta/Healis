# Архитектура системы

## Высокоуровневый обзор

Классическая трёхслойная архитектура:

Client (React SPA) ←→ Backend API (Node.js/Express + Socket.IO) ←→ PostgreSQL

Дополнительно: файловое хранилище (S3/Supabase), кеш/adapter (Redis) и интеграционные адаптеры для внешних API.

## Компоненты и взаимодействие

1. Frontend (Client)
   - SPA на React, отвечает за рендер интерфейса, отображение досок, локальное состояние и оптимистичные обновления.
   - Использует HTTP (REST) для CRUD операций и Socket.IO для real-time синхронизации.

2. Backend API
   - REST API на Express для управления сущностями (users, workspaces, materials, events).
   - Socket.IO сервер для синхронизации изменений досок и рассылки уведомлений в реальном времени.
   - Авторизация через middleware (JWT), проверка прав и валидация входящих данных.
   - ORM: Sequelize для работы с Postgres, миграции и сиды.

3. Database
   - PostgreSQL (JSONB для сложных полей, таких как состояние доски).
   - Индексы по FK и полям поиска.

4. Хранилище файлов
   - S3-совместимое хранилище или Supabase Storage для медиа и вложений.
   - При загрузке файлов возвращаются публичные/приватные URL в метаданных.

5. Интеграционные адаптеры
   - Отдельные модули для SSO и внешних API (расписание, списки групп).
   - Синхронизация может быть event-driven (webhooks) или polling.

6. Кэш и масштабирование real-time
   - Redis рекомендуется для:
     - Adapter Socket.IO (горизонтальное масштабирование real-time)
     - Кэширование интеграционных ответов
     - Очередей для фоновых задач

## Потоки данных

- Аутентификация: SSO → Backend (создание/обновление профиля) → JWT клиенту.
- Работа с доской: клиент отправляет изменения → сервер валидирует и сохраняет → сервер рассылает изменения через Socket.IO другим участникам комнаты.
- Загрузка файлов: клиент → /uploads → файловое хранилище; сервер сохраняет метаданные (file_url) в БД.
- Синхронизация расписания: интеграционный модуль периодически вызывает API вуза или получает webhook и обновляет локальные сущности.

## Масштабирование и рекомендации

- Горизонтальное масштабирование backend: несколько инстансов за load-balancer; для Socket.IO нужен Redis adapter и sticky-sessions на LB.
- Отделение чтения и записи БД: master-replicas для снижения нагрузки чтения.
- Отдельные сервисы/микросервисы для тяжёлых фоновых задач (обработка медиа, уведомления, интеграции).
- CDN для фронтенда и статического контента.

## Надёжность и резервирование

- Резервные копии Postgres и проверка restore.
- Мониторинг метрик (CPU, latency, error rates) и alerting.
- Тестирование отказоустойчивости и сценариев восстановления.

## Интеграция с LLM (план)

- LLM-сервис (например, Ollama) может быть подключён как отдельный сервис, принимающий содержимое досок для анализа:
  - Архитектура: Backend → LLM (сервис) (с фильтрацией конфиденциальных данных).
  - Требования: контроль доступа, приватность, квоты и throttling, логирование запросов и ответов.

## Диаграмма (упрощённая)

Client (React) -- HTTPS --> API (Express + Socket.IO) -- SQL --> Postgres
                             |
                             --> S3 / Storage
                             |
                             --> External Integrations (SSO, Univ APIs)

## Вывод

Архитектура Healis спроектирована для быстрой интерактивной работы и постепенного масштабирования: real-time синхронизация локальных рабочих пространств, безопасные интеграции с университетами и возможность добавления аналитики/ИИ как отдельного слоя.
