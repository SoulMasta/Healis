# Интеграция с университетскими системами

## Цели интеграции

Интеграция служит для получения академических данных (расписание, сведения о курсах, учебные материалы и т.д.) и аутентификации пользователей через корпоративный SSO университета. Healis не изменяет данные в системах университета — доступ ограничен режимом read-only.

## Аутентификация (SSO)

- Поддерживаемые подходы: SAML, OAuth2 / OpenID Connect (в зависимости от возможностей вуза).
- Поток: пользователь переходит на SSO провайдера → после успешного входа Healis получает assertion / токен → создаётся/обновляется профиль пользователя в Healis.
- Внутри сервиса для локальной авторизации используется JWT, привязанный к учётной записи пользователя (внутренние API).
- Рекомендуется настроить короткий TTL для локальных токенов и проверять валидность SSO-учётных данных при изменениях прав.

## Доступ к данным

- Типы данных, которые могут поставляться интеграциями:
  - Расписание (занятия, аудитории, преподаватели)
  - Списки студентов/групп (для отображения релевантных расписаний)
  - Учебные материалы (ссылки на ресурсы или метаданные; сами файлы обычно хранятся в файловом хранилище)
  - Оповещения / объявления (опционально)

- Протоколы: REST API, SOAP, или проприетарные API вуза. Для каждого интеграционного адаптера реализуется уровень преобразования в внутреннюю модель Healis.
- Доступ осуществляется через публичные API университета — прямого подключения к БД университета не требуется и не практикуется.

## Контракты и схемы

- Каждый интеграционный модуль должен описывать входной контракт (поля, форматы дат, идентификаторы групп).
- При проектировании адаптера учитывать пагинацию, throttling и retry при недоступности внешнего API.

## Безопасность интеграций

- Шифрование каналов (HTTPS/TLS) обязательно.
- Хранение секретов (client_id, client_secret) — в защищённых переменных окружения / секретном хранилище (Vault).
- Логирование вызовов интеграции с маскированием персональных данных.
- Ограничение прав: сервис интеграции должен работать от сервисной учётной записи с минимально необходимыми правами.

## Обработка ошибок и отказоустойчивость

- Кэширование результатов (например, расписания) с временем жизни, чтобы снизить нагрузку на внешние API.
- Фолбэки: при недоступности интеграции показывать старые кэшированные данные и информировать пользователя.
- Система должна корректно обрабатывать частичные ответов и пропуски полей.

## Примеры интеграционных сценариев

- Аутентификация через OpenID Connect: перенаправление пользователя, получение ID Token, валидация подписи, создание локального аккаунта.
- Получение расписания по REST: периодическая синхронизация через cron-джоб или webhook от вуза.

## Замечания по реализации

- Поддерживать адаптеры как отдельные модули/микросервисы или плагины, чтобы легко добавлять новые вузы.
- Документировать схемы данных и ошибки для каждого адаптера.
