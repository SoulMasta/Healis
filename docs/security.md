# Безопасность

Этот документ описывает ключевые меры безопасности, применяемые в Healis, и рекомендации по дальнейшему улучшению защиты.

## Аутентификация и авторизация

- Первичный способ входа — университетское SSO (SAML или OpenID Connect). После успешной аутентификации создаётся локальная запись пользователя и выдаётся JWT для доступа к internal API.
- Пароли (если используются локальные учётные записи) хранятся в виде хэшей с использованием bcrypt (с достаточным work factor).
- Ролевая модель: student, teacher, admin. Контроль доступа реализуется на уровне middleware в API и авторизации в запросах к базе данных.

## Транспортный уровень

- Все внешние соединения должны быть по HTTPS/TLS (рекомендуется TLS 1.2+).
- Внутренние подключения между сервисами (если есть) также следует защищать и помещать в приватную сеть.

## Хранение секретов

- Секреты (JWT_SECRET, SSO client secrets, ключи доступа к хранилищу) не хранятся в репозитории. Использовать переменные окружения, CI/CD secrets или менеджер секретов (Vault, cloud secrets).

## Защита данных и приватность

- Хранить минимально необходимые персональные данные.
- Чувствительные поля логов маскировать (например, токены, пароли).
- Реализовать политику удаления/ретенции данных (например, удаление неактивных аккаунтов по запросу).

## Валидация и защита API

- Валидация всех входящих данных (использовать схемы Joi/Zod/express-validator).
- Ограничение размера загрузок и проверка MIME-типов вложений.
- Защита от SQL-инъекций — использовать ORM (Sequelize) с параметризованными запросами.
- Ограничение частоты запросов (rate limiting) на критичные endpoints (логин, upload).
- CORS настроен строго для доверенных доменов.

## Реагирование и логирование

- Логирование: аудит доступа к API, ошибки, попытки входа. Логи отправлять в централизованную систему (ELK, Cloud Logging).
- Настроить оповещения на аномалии (частые ошибки 5xx, всплески попыток логина).
- План реагирования на инциденты (инструкция по расследованию и восстановлению).

## Безопасность интеграций

- Для интеграций с внешними API (университеты) использовать сервисные аккаунты с минимальными правами.
- Шифровать и защищать каналы передачи данных.

## Безопасность real-time

- WebSocket/Socket.IO соединения проверяются на этапе handshake (валидация JWT).
- Ограничивать размер и частоту сообщений по каналам.
- При масштабировании через Redis adapter — защита Redis соединений (пароли, приватные сети).

## Рекомендации по hardening

- Регулярные обновления зависимостей, мониторинг известных уязвимостей (Dependabot/Snyk).
- Периодические pentest/сканирование уязвимостей.
- Ограничение прав сервисных учётных записей (least privilege).

## Соответствие и конфиденциальность

- При интеграции с персональными данными студентов обратить внимание на местные требования по защите данных и хранению персональной информации.
